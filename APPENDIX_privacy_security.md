# Sidekick Appendix: Privacy and Security

Sidekick captures unusually sensitive work context: files, tickets, PRs, messages, meetings, docs, and the user's own memory. The system therefore uses a local-first boundary, explicit capture allowlists, redaction before any cloud LLM call, provenance everywhere, and inspectable deletion controls.

Assumptions:
- Sidekick is single-user and local-first, not a team or multi-tenant product.
- Source integrations are read-only unless the user explicitly approves a future capability.
- Cloud LLM use is optional; local-model mode is available for privacy-sensitive work.
- The Local Companion is the only process allowed to hold long-lived credentials.

## Data Lifecycle

| Stage | What happens | Privacy control |
|---|---|---|
| 1. Capture | Surfaces and connectors observe allowed work signals. | Source-level capture allowlist; disabled by default per source. |
| 2. Normalize | Raw signals are mapped into `WorkContextEvent`. | Raw payloads are not retained unless explicitly enabled for debugging. |
| 3. Redact | Sensitive content is minimized before storage or model use. | PII and secrets are removed or replaced with stable placeholders. |
| 4. Store | Events, commitments, lessons, and memory records are persisted locally. | SQLite database and semantic memory encrypted at rest. |
| 5. Retrieve | Capabilities retrieve a small relevant set. | Retrieval avoids dumping the whole timeline into prompts. |
| 6. Optional cloud LLM | The LLM Reasoning Core sends only redacted context when cloud mode is enabled. | User-visible setting; local-model mode bypasses cloud calls. |
| 7. Surface | Interaction / Attention Policy decides ambient, propose, or never. | Every surfaced item includes provenance and can be dismissed. |
| 8. Delete | User deletes events, commitments, lessons, memory records, or whole source history. | Delete propagates to derived memory where provenance depends on deleted data. |

## Redaction Before Cloud Calls

Before a cloud LLM receives context, the **Context Capture Layer** and **LLM Reasoning Core** apply a redaction pass. The goal is not perfect anonymization; it is data minimization plus explicit consent for any remaining context.

| Data class | Treatment before cloud LLM |
|---|---|
| OAuth tokens, API keys, cookies, session IDs | Always stripped; never sent. |
| Secrets in files or logs | Pattern-detected and replaced with `[SECRET]`. |
| Personal email addresses and phone numbers | Replaced with stable local aliases such as `person_3`. |
| Private message bodies | Summarized locally when possible; only short redacted summary sent. |
| Source URLs | Sent only when needed for provenance; private path segments can be hashed. |
| File contents | Not sent by default. Snippets require explicit user approval or source-specific allowlist. |
| Calendar details | Titles summarized; attendee lists minimized. |
| Meeting notes | Action-item summaries preferred over full transcript text. |
| Local file paths | Reduced to repo-relative paths where possible. |

Local-model mode keeps all reasoning on the machine. In that mode, redaction still runs before prompt construction because the same minimization protects logs, traces, and future debugging exports.

## Threat Model

| Scenario | What could leak | Mitigations |
|---|---|---|
| Malicious extension update | Captured page context, local API requests, UI output. | Thin Surfaces only; Local Companion requires localhost auth token; extension permissions are narrow; source allowlist is visible and revocable. |
| Compromised OAuth token | Read access to GitHub, Slack, Jira, mail, calendar, or docs. | Store tokens in OS keychain; request least-privilege scopes; support token revocation; health checks detect auth drift. |
| Cloud LLM logging or retention | Redacted summaries, project names, source refs, commitment text. | Cloud mode opt-in; local-model mode available; redaction before send; provider configured for no-training/no-retention where available; cite-or-suppress reduces broad prompt dumps. |
| Local database theft | Timeline, commitments, lessons, semantic memory. | Encrypt at rest; use OS account boundary; keep database under user profile; avoid raw payload retention; support full export/delete. |
| Prompt injection from docs or messages | Malicious text tries to override policies or exfiltrate memory. | Treat source content as untrusted data; tool calls behind schemas; no autonomous actions; no source write access; cite-or-suppress and confidence thresholds. |
| Overcapture by a valid connector | Personal or unrelated work context enters memory. | Capture allowlist per source/project/channel/repo; `origin: work | personal`; inspect/delete UX; retention policies. |
| Localhost API abuse | Another local process posts fake events or reads resume state. | Bind to loopback only; require bearer token generated by Local Companion; reject unauthenticated requests; rate-limit local clients. |
| Stale or wrong provenance | Surfaced claim points to the wrong source or unsupported inference. | Store provenance with events and derived entities; suppress low-confidence claims; show source refs with every surfaced item. |

## OAuth Token Handling

- The **Local Companion** owns OAuth flows and long-lived credentials.
- Tokens are stored in the OS keychain, not in plaintext config files.
- Surfaces receive only a short-lived localhost session token for talking to the Local Companion.
- Connector scopes are least-privilege and read-only:
  - GitHub/GitLab: PRs, reviews, mentions, repo metadata.
  - Jira/Linear: assigned tickets, comments, status.
  - Slack/Teams: explicitly allowed channels, DMs, mentions.
  - Outlook/Gmail: selected labels or inbox categories.
  - Calendar: availability and selected calendar events.
  - Meeting AI and Docs: selected workspaces or documents.
- Revoking a connector removes its token, stops new capture, and marks related local data for user-selected retention or deletion.

## Encryption at Rest

The **State + Timeline Store** is local, but local does not mean unprotected.

- SQLite database file is encrypted at rest in the production design.
- Semantic memory and vector index entries use the same encryption boundary.
- Encryption keys are protected by the OS keychain or secure enclave where available.
- Backups are opt-in and inherit the same encryption policy.
- Debug logs never include raw OAuth tokens, raw message bodies, or full prompts by default.

For the complete local implementation in `sidekick/`, encryption is documented as future work; the implementation focuses on authenticated local capture, redaction, SQLite timeline storage, inspect/delete behavior, and all five capabilities over the shared local brain.

## Capture Allowlist

Capture is explicit and inspectable. The default posture is "capture nothing until enabled."

| Level | Example setting |
|---|---|
| Source | Enable GitHub, leave Slack disabled. |
| Account | Enable work GitHub account, exclude personal account. |
| Project | Enable `acme/payments`, exclude unrelated repos. |
| Channel | Enable `#payments-eng`, exclude social or private channels. |
| Kind | Capture `reviewing_pr` and `edited_file`, skip raw file contents. |
| Time | Disable capture during personal hours or calendar focus blocks. |

The allowlist is enforced before normalization. If a signal is not allowed, it is dropped and does not enter the timeline.

## Inspect and Delete UX

Inspectable and reversible is a core product principle, not an admin feature.

- **Event viewer:** show captured `WorkContextEvent` rows with source, kind, summary, confidence, and provenance.
- **Derived entity viewer:** show `Commitment`, `Lesson`, and `MemoryRecord` items with the source events that produced them.
- **Delete one item:** removes the selected entity and flags dependent derived records for recomputation or removal.
- **Delete source history:** removes all events and derived data for a source, repo, channel, project, or date range.
- **Correct memory:** lets the user edit or demote a semantic memory record; preference updates require confirmation.
- **Cloud-call preview:** shows the redacted prompt context before first cloud LLM use for a source.

## Security Posture by Principle

| Principle | Security implication |
|---|---|
| Local-first | Work context stays on the user's machine unless explicitly sent to a cloud model. |
| Propose-don't-act | Prompt injection cannot trigger autonomous source writes. |
| Provenance everywhere | Surfaced claims are traceable and unsupported claims are suppressed. |
| Pluggable edges | A compromised connector can be disabled without rewriting the core. |
| Graceful degradation | Cloud or connector failures do not require unsafe fallback behavior. |
| Inspectable and reversible | The user can audit, correct, and delete captured memory. |
